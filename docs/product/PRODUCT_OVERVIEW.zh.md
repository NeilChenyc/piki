# Piki 总体产品文档

## 1. 产品定位

Piki 是一个面向高密度知识工作者的本地优先个人记忆系统。

它不是普通笔记工具，也不是简单 RAG 搜索工具，而是基于 LLM Wiki 模式，把用户分散的资料、对话、笔记和思考持续编译成一个可维护、可回忆、可追溯的个人 Wiki。

一句话定位：

> Piki 帮助资料很多、经常复用信息、但不想再手动维护知识库的人，把零散输入变成长期可调用的个人记忆。

## 2. 目标客群

### 核心客群：知识工作者

- AI 产品经理
- 独立开发者
- 工程师 / AI 开发者
- 研究员 / 咨询顾问
- 创业者
- 投资 / 行业分析人员
- 技术博主和专业内容创作者

这类用户的共同特征是：信息输入多、复用频率高、记忆损耗成本高。

### 次级客群：学生与研究型学习者

- 研究生
- 论文写作者
- 考试复习用户
- 留学生
- 课程 / 项目学习者

这类用户体量大、传播快，但付费敏感度更高，更适合作为增长入口。

### 延展客群：内容创作者

- 博主
- 播客 / 视频创作者
- 写作者
- 课程创作者

他们需要把素材、观点、案例、脚本和灵感沉淀成可复用内容资产。

### 后期客群：个人事务管理用户

- 医疗记录管理
- 保险 / 签证 / 租房资料管理
- 家庭维修记录
- 旅行攻略
- 购物研究
- 生活决策记录

这类人群广，但主动维护知识库意愿较弱，不适合作为 MVP 第一目标。

## 3. 核心痛点

- 信息过载：网页、PDF、聊天、文档、视频、笔记持续堆积。
- 收藏失效：收藏不等于记住，很多资料存了之后再也不会被调用。
- 搜索不够：搜索只能找到片段，不能帮助用户形成长期理解。
- 维护成本高：Notion、Obsidian 等工具强大，但长期整理成本太高。
- AI 上下文断裂：每次问 AI 都要重新解释背景，个人长期上下文无法自然延续。
- 知识复用低：写 PRD、报告、文章、研究材料时，经常重复整理同一批背景信息。
- 资料不可追溯：AI 回答常常脱离来源，用户难以确认结论从哪里来。

## 4. 用户核心需求

### 记住

- 能快速收集资料，不打断当前工作流。
- 能把原始资料保留下来，避免被 AI 摘要污染。
- 能自动整理来源、概念、实体、领域和相关结论。
- 能识别冲突、不确定内容和需要复审的信息。
- 能让知识库随着资料和对话持续增长。

### 回忆

- 能用自然语言问回自己的知识库。
- 能找到过去看过、想过、讨论过的内容。
- 能获得带引用、可追溯的回答。
- 能通过概念、实体、领域、链接关系重新发现旧知识。
- 能把有价值的回答继续保存回知识库。

### 维护

- 能降低人工整理成本。
- 能看到知识库健康状态。
- 能发现孤儿页、断链、重复概念、过期内容和知识缺口。
- 能把低置信度和敏感内容交给用户审核，而不是自动写死。

## 5. 产品核心假设

- 一部分高密度知识用户并不缺笔记工具，而是缺“自动维护长期知识资产”的系统。
- 个人知识库的价值不在于存了多少资料，而在于未来能否被可靠回忆和复用。
- LLM 的最佳角色不是替用户“记笔记”，而是维护一个可读、可追溯、可增长的个人 Wiki。
- 用户需要保留控制权：原始资料不可变，重要写入可审核，结论可追溯。
- MVP 应先服务 5%-10% 信息密度最高、痛点最强的人，而不是一开始追求大众市场。

## 6. 产品形态

Piki 的底层是一个本地 Markdown vault：

- `raw/` 保存原始资料。
- `wiki/` 保存 LLM 编译后的知识页面。
- `AGENTS.md` 定义 agent 如何维护知识库。
- `index.md` 和 `log.md` 帮助用户和 agent 导航、追踪变化。

Piki 的上层是一个个人记忆工作台：

- 收集资料。
- 触发 ingest。
- 审核不确定内容。
- 浏览知识网络。
- 向知识库提问。
- 将高价值对话内容保存回 wiki。
- 检查知识库健康状态。

在 MVP 阶段，Piki 的 agent 层优先直接接入 Codex CLI，本地 Markdown vault 仍然是真正的数据底座。Piki 自己的客户端 UI 先承担会话承载、命令触发、文件预览、变更确认、审核队列和结果展示，不在第一阶段自研完整 agent runtime。

## 7. MVP 需求范围

MVP 应优先覆盖以下需求：

- 本地 vault 初始化。
- 原始资料导入和 inbox。
- Agent 按 LLM Wiki 规则维护知识库。
- MVP 阶段 agent 层优先复用 Codex CLI。
- Analyze -> Generate 两阶段 ingest。
- Review queue，处理不确定、冲突和敏感内容。
- 基础中文友好检索。
- 基于 wiki 的问答和引用。
- Save to Wiki，将高价值对话作为 source 保存，并分类编译进 wiki。
- 基础维护检查。
- 简易 Mac 客户端，用于浏览、提问、审核和维护。

MVP 客户端不追求把 CLI 伪装成终端，而是要把它渲染成面向知识库工作的 UI：

- 对话消息流
- 命令与 operation 展示
- 文件 diff / 引用 / 预览
- ingest / review / maintenance 状态
- source、wiki 页面与生成文档的联动查看

## 8. MVP 不做

- 不做大众化笔记 App。
- 不做账户系统。
- 不做云同步。
- 不做多人协作。
- 不做移动端。
- 不一开始支持所有文件格式。
- 不把向量数据库作为核心前置依赖。
- 不做全自动无审核 ingest 一切。
- 不把聊天历史默认当作长期记忆。

## 9. 差异化

- 相比普通笔记工具：Piki 重点不是写笔记，而是持续维护知识结构。
- 相比 Obsidian：Piki 让 LLM 承担整理、链接、更新和检查成本。
- 相比 RAG：Piki 不只是临时检索 chunks，而是构建持续增长的编译知识层。
- 相比普通 AI 聊天：Piki 有长期记忆、来源追踪和可审核的本地 Wiki。

## 10. 成功标准

- 用户愿意持续把资料放进 Piki。
- 用户能在需要时问回自己的旧知识。
- 用户能信任回答，因为答案带引用且可追溯。
- 用户能把高价值对话内容沉淀成长期记忆。
- 用户明显减少重复整理资料的时间。
- 知识库不会因为自动化写入而变脏。
- 即使不用 App，用户仍能直接打开 Markdown vault 阅读和迁移数据。

## 11. MVP 首选切入

Piki 的 MVP 不应该面向“所有想记笔记的人”，而应该优先服务：

> AI / 产品 / 研究型知识工作者，以及独立开发者、技术博主、咨询和行业研究人员。

他们资料密度高，复用需求强，已经被现有笔记、收藏夹、文件夹和 AI 上下文断裂折磨过，更容易理解 Piki 的价值。
