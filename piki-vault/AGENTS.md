# Piki LLM Wiki 协议

本文件是这个 vault 的 schema。它参考 Andrej Karpathy 在 GitHub Gist 发布的 `llm-wiki.md`，把 LLM Wiki 的通用模式实例化为 Piki 当前使用的中文知识库协议。

这个 vault 不是普通笔记目录，而是一个由 Agent 持续维护的中文 LLM Wiki。
用户负责提供来源、提出问题、决定方向；Agent 负责整理、编译、交叉链接、更新索引、维护日志和保持一致性。

## 1. 核心理念

- 不要把这个 vault 当成“每次问答都重新检索原文”的 RAG 容器。
- 要把它当成“持续编译中的知识库”：
  Agent 从 `raw/` 读取来源，把知识沉淀到 `wiki/`，让知识随着每次 ingest、query、lint 持续累积。
- 默认优先从 `wiki/` 回答，而不是每次重新通读所有原始来源。
- `wiki/` 是持久化、可复用、会不断增厚的中间层；回答、比较、总结、冲突标记，只要有长期价值，都应优先沉淀在这里，而不是只留在聊天记录里。

## 2. 三层结构

这个 vault 严格分成三层：

### 2.1 `raw/`：原始来源层

- `raw/` 存放来源材料和来源归档。
- `raw/inbox/` 是待处理来源的入口区。
- `raw/assets/` 存放下载或归档后的附件、原始文件、图片等。
- `raw/sources/` 存放来源的 canonical Markdown 归档页。
- 原始来源层的原则是“归档优先、轻改写、不可随意重写”。
- 不要为了回答问题直接把 `raw/` 当作长期工作区。
- 不要为了方便而覆盖、重写、删改原始来源内容；若确需调整，应优先新增归档或新增说明，而不是篡改历史来源。

### 2.2 `wiki/`：编译后的知识层

- `wiki/` 是 Agent 维护的主体知识库。
- 这里存放经过整理、抽象、链接和综合后的页面。
- 当前主要目录约定如下：
  - `wiki/sources/`：每个来源对应的 wiki 来源页
  - `wiki/entities/`：人物、组织、产品、项目等实体页
  - `wiki/concepts/`：概念、方法、机制、术语页
  - `wiki/domains/`：主题域、知识域、长期主题页
  - `wiki/synthesis/`：综合分析、比较、结论、阶段性沉淀
- `wiki/` 是主要回答层。正常 query 应优先读取这里。
- 允许一个来源影响多个 wiki 页面；不要把知识只塞进单个来源页。

### 2.3 `AGENTS.md`：规则层

- `AGENTS.md` 定义目录结构、页面职责、工作流和写作约束。
- `AGENTS.md` 只读。
- 后续如果用户明确要求更新协议，可以调整本文件，但默认任何任务都不能改写它。

### 2.4 `system/`：系统状态层

- `system/` 不属于人类主要阅读的知识层。
- 它用于保存运行时所需的内部状态、索引和 manifest。
- 当前最重要的文件是 `system/source_manifest.json`，用于记录来源哈希、归档路径和复用关系。
- `system/` 的内容应被视为操作性元数据，而不是 wiki 知识页面。
- query 默认不应把 `system/` 当成主要知识来源。
- 除非任务明确与 intake、manifest、扫描或系统维护有关，否则不要优先读取、展示或改写 `system/`。
- `system/` 可以被系统流程更新，但不应和 `wiki/` 页面混为一谈。

## 3. 中文约束

这是本项目的强约束，优先级高于通用模板偏好：

- 组内文档默认使用中文。
- 至少标题和主要内容必须是中文。
- 新建 wiki 页时，优先使用中文标题。
- 页面摘要、结论、关系说明、综合判断应使用中文。
- 如果原始术语是英文，可以保留英文原词，但应给出中文解释或中文上下文。
- 文件名允许为了兼容来源或 slug 混用英文、数字、哈希，但页面标题和正文主体仍应以中文为主。
- 引用原文时可以保留少量英文原句，但不要让整页主体退化成英文摘抄。

## 4. 页面写作原则

- 优先写“可被未来复用”的页面，不写一次性聊天腔内容。
- 页面应强调：
  - 这是什么
  - 它和哪些页面相关
  - 它和旧认识相比新增了什么
  - 是否存在冲突、不确定性、时效性问题
- 看到有价值的关系时，应补充 wikilink。
- 当新来源推翻旧判断时，要更新相关 wiki 页面并明确标注变化，而不是把冲突默默覆盖掉。
- 如果信息不足，不要编造；要明确写出“不确定”“待补充来源”“需要复核”。

## 5. `index.md` 与 `log.md`

这两个文件是 LLM Wiki 的关键导航层，职责不同，不要混用。

### 5.1 `wiki/index.md`

- `wiki/index.md` 是内容导向的目录。
- 它应按类别列出重要页面，并给出一句话摘要。
- query 时，优先先读 `wiki/index.md`，再钻取相关页面。
- ingest 新来源后，应同步更新 `wiki/index.md`。
- 新建了重要页面但没有进入 `wiki/index.md`，通常算缺失维护。

### 5.2 `wiki/log.md`

- `wiki/log.md` 是时间导向的追加日志。
- 它记录 ingest、重要 query 沉淀、lint、修复、回退等事件。
- 日志是 append-only，默认只追加，不重写旧记录。
- 标题格式保持可 grep、可解析：

`## [YYYY-MM-DD] 操作 | 简短标题`

- 单条日志应尽量写清：
  - 来源或任务是什么
  - 改了哪些路径
  - 做了什么类型的知识更新

## 6. Query 工作流

用户提问时，默认遵循以下顺序：

1. 先读 `wiki/index.md`，定位相关页面。
2. 再只读取和问题直接相关的 `wiki/` 页面。
3. 优先基于 `wiki/` 回答，而不是重扫所有 `raw/`。
4. 回答中尽量引用具体页面路径或页面名。
5. 如果 `wiki/` 已经足够，不要回退到原始来源。
6. 如果 `wiki/` 明显不足，可以说明知识缺口；仅在必要时回看少量相关来源。

额外要求：

- 普通 query 默认不写入。
- 但如果用户明确要求“把这次分析沉淀到知识库”“保存结论”“生成综合页”，可以把高价值回答写入 `wiki/synthesis/` 或相关页面。
- 新沉淀内容必须可复用，不能只是聊天原话转存。

## 7. Ingest 工作流

ingest 的目标不是“把文件放进仓库”，而是“把来源编译进 wiki”。

标准顺序：

1. 获取并确认来源材料。
2. 在 `raw/` 层完成来源归档：
   - 原始文件进入 `raw/assets/` 或保留在受控来源路径
   - canonical 来源页进入 `raw/sources/`
3. 产出或更新 `wiki/sources/` 页面，形成该来源的可读摘要页。
4. 按需要更新相关的：
   - `wiki/entities/`
   - `wiki/concepts/`
   - `wiki/domains/`
   - `wiki/synthesis/`
5. 更新 `wiki/index.md`。
6. 在 `wiki/log.md` 追加记录。

ingest 原则：

- 一个新来源通常不应只改一页来源页就结束。
- 要把知识整合进现有 wiki，而不是把新知识孤立堆积。
- 若新来源与旧页面矛盾，要明确标出冲突或更新判断。
- 若某个重要概念频繁被提及但还没有页面，应考虑补页。
- 若来源价值很低、重复度很高，可以少写，但仍应保持索引和日志的一致性。

## 8. Lint / 健康检查工作流

lint 的目标是健康检查与定向修复，不是趁机大改全库。

优先关注：

- 断裂链接
- 孤立页
- 缺失索引项
- 重复标题
- 内容过薄页面
- 过期或被新来源推翻但未更新的判断
- 提及频繁但尚未成页的重要概念
- 缺少必要交叉链接的页面

lint 原则：

- 如果系统提供了结构化 lint 结果，应先基于结构化结果行动。
- lint 过程中不要顺手做无关的大规模扩写、重构或风格统一。
- 修复应聚焦于问题直接涉及的页面，以及必要的 `wiki/index.md` / `wiki/log.md`。
- 如果发现真正的知识空洞，可以在结果中提出建议，但不要把“发现空洞”误当成“必须立即扩写全库”。

## 9. 页面关系与链接约定

- 重要名词首次出现后，若库中已有对应页面，优先补 `[[wikilink]]`。
- 新建页面时，要主动补充到至少一个上游或同层页面的链接关系，避免孤立。
- `wiki/sources/` 页面主要回答“这个来源说了什么”。
- `wiki/entities/` 页面主要回答“这个实体是谁、做了什么、和哪些来源/概念相关”。
- `wiki/concepts/` 页面主要回答“这个概念是什么、与哪些实体/来源/域相关”。
- `wiki/domains/` 页面主要回答“这个知识域的核心结构与关键页面有哪些”。
- `wiki/synthesis/` 页面主要承载跨来源、跨页面的综合判断。

## 10. 人与 Agent 的分工

- 用户负责：
  - 提供来源
  - 决定关注重点
  - 判断哪些结论值得长期沉淀
- Agent 负责：
  - 阅读与提炼
  - 分类与归档
  - 更新交叉链接
  - 保持索引与日志同步
  - 发现冲突、缺口和过期内容

默认情况下，用户不需要手工维护 wiki 结构细节；Agent 应承担大部分“知识库记账工作”。

## 11. 针对当前 Piki 测试库的特别约束

- 这是轻量测试 vault，不是正式生产知识库。
- 在保持 LLM Wiki 结构完整的前提下，优先追求：
  - 上下文小
  - 响应快
  - 行为可预测
  - 变更可验证
- 不要为了“看起来更完整”而做无边界扩写。
- 普通 query 不应触发大范围写入。
- 无用户明确要求时，不要批量重写大量现有页面。

## 12. 成功标准

一个好的 Agent 行为应满足：

- 回答问题时，主要依赖已经编译好的 wiki。
- ingest 新来源后，知识能进入多个合适页面，而不是只留下孤立摘要。
- `wiki/index.md` 和 `wiki/log.md` 始终可用、可信、同步。
- 页面之间有持续增强的中文交叉链接。
- 冲突、不确定性、时效性问题被明确标注，而不是被掩盖。
- 整个 vault 越用越像一个持续增长的中文知识库，而不是一堆临时文件。
